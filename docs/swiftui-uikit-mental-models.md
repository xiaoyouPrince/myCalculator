# SwiftUI 真的比 UIKit 简单吗？一次日历视图 Bug 的复盘

## 开头

最近在做一个业余工具项目：一个用 SwiftUI 写的 macOS 工时日历。功能并不复杂，主要是按日、周、月、年记录上下班时间，并计算工时和加班。

但在修复月视图的一个显示问题时，我重新意识到一件事：SwiftUI 的代码确实更短，但它并没有消灭 UI 系统本身的复杂度。很多在 UIKit 中显式存在的生命周期、复用和刷新控制，在 SwiftUI 中被隐藏到了 diff、identity 和状态保留机制里。

这篇文章不是为了证明 SwiftUI 或 UIKit 谁更好，而是想从一次真实踩坑出发，梳理两套 UI 框架背后的心智模型差异。

## 问题背景

这个 App 的月视图需要展示真实日历网格。以 2026 年 7 月为例：

- 7/1 到 7/31 属于当前月，应使用正常深色。
- 6/29、6/30、8/1、8/2 只是为了补齐日历网格，应使用淡化灰色。
- 如果今天不属于当前展示月份，就不应该出现今日高亮。

用户的操作路线很简单：

- 启动 App，默认展示 2026 年 6 月视图。
- 点击月视图右上角下一月按钮，进入 2026 年 7 月视图。
- 检查第一周和最后一周的跨月日期颜色。

第一次修复后，我只做了编译验证，结果用户截图显示仍然是错的：

- 6/30 在 7 月视图中仍然被蓝色高亮。
- 7/1 到 7/5 反而被淡化。
- 左侧日期选择器还一度把年份解析成异常值，出现了类似 `Jul 30, 4` 的显示。

这个问题非常适合作为 SwiftUI 与 UIKit 对比的切入点。

## UIKit 的复杂度在明处

UIKit 是命令式 UI。你创建 view，设置属性，响应生命周期，主动告诉界面什么时候刷新。

在 UIKit 中，列表或网格复用是一个显式概念：

- 你知道什么时候调用 `dequeueReusableCell`。
- 你知道什么时候进入 `cellForItemAt` 或 `cellForRowAt`。
- 你知道什么时候需要 `prepareForReuse`。
- 你知道什么时候调用 `reloadData`、`reloadItems` 或 `performBatchUpdates`。

UIKit 的麻烦在于代码多。你需要写 datasource、delegate、cell 配置、复用清理、布局刷新。但它的好处也很直接：出问题时，排查路径通常比较清晰。

如果一个日历 cell 的颜色不对，你大概率会去看 cell 配置逻辑：

- 当前 cell 绑定的是哪个 model。
- `isCurrentMonth` 是否正确。
- cell 复用时是否清掉旧颜色。
- `reload` 是否真的触发。

UIKit 要求开发者管理对象。对象生命周期、复用边界和刷新时机虽然繁琐，但都在明面上。

## SwiftUI 的复杂度在暗处

SwiftUI 是声明式 UI。你描述当前状态下 UI 应该长什么样，框架负责根据状态变化更新界面。

这让代码看起来很自然：

- 当前月日期使用正常颜色。
- 非当前月日期使用灰色。
- 今天使用浅蓝背景。

但真正决定界面如何更新的，不只是这几行条件判断。SwiftUI 还会综合考虑：

- `ForEach` 的 `id`。
- 父子视图的 identity。
- `@State` 与 view identity 的绑定关系。
- `LazyVGrid`、`List`、`LazyVStack` 的懒加载和复用。
- 新旧 view tree 的 diff 结果。
- 是否使用 `.id(...)` 强制重建子树。

这也是 SwiftUI 容易让人产生错觉的地方。我们以为自己写的是“状态变了，UI 自动正确刷新”，但实际上 SwiftUI 做的是“状态变了，框架根据 identity 判断哪些视图应该延续，哪些视图应该重建”。

SwiftUI 的 View 是值描述，不是传统意义上的 view object。真正被保留的往往是状态、identity 和渲染子树。

## 这次 Bug 的真正原因

问题一开始看起来像日期算法错误。于是我先修了月份判断逻辑，把是否属于当前月改成按年/月组件判断。

这种判断本身是正确的：

- 单元格日期的 year 等于当前展示月份的 year。
- 单元格日期的 month 等于当前展示月份的 month。

代码级验证也能证明 2026 年 7 月的数据是对的：

- `06/29`、`06/30` 是 outside。
- `07/01` 到 `07/31` 是 current。
- `08/01`、`08/02` 是 outside。

但截图仍然错。这说明问题不只在数据计算层，而在渲染层。

月视图当时使用了 `LazyVGrid`，并且按行下标作为 `ForEach` 的 identity。切换月份后，第一行仍然是第一行，第二行仍然是第二行。SwiftUI 有理由认为这些位置上的视图可以延续。

可是业务语义已经变了：

- 6 月第一周里的 6/30 是当前月。
- 7 月第一周里的 6/30 是非当前月。
- 7 月第一周里的 7/1 到 7/5 才是当前月。

如果 SwiftUI 复用了旧月份首周的渲染状态，就会出现“数据已经变了，但显示还像旧月份”的问题。

最终修复不是只改日期算法，而是同时处理 identity 和显示上下文：

- `MonthDayCell` 根据当前展示月份和单元格日期实时计算 `isInDisplayedMonth`。
- 整个月网格增加月份级 `.id(monthIdentity)`。
- 月份变化时强制重建网格，避免上一月首周状态残留。
- 当前月和非当前月使用明确颜色，不只依赖 `.primary` 和 `.secondary`。

这次问题的核心教训是：在 SwiftUI 中，数据正确不一定代表 UI 会按你的直觉重新渲染。identity 设计不对，旧状态可能会以非常隐蔽的方式留下来。

## SwiftUI 不是没有复用，而是换了一种复用方式

UIKit 里的复用很直观。你写 `dequeueReusableCell`，你知道 cell 被复用了。

SwiftUI 里你不写 dequeue，但不代表没有复用。复用只是换成了另一种形式：

- view tree diff。
- identity 复用。
- lazy container 延迟生成。
- 状态与 identity 绑定。

所以 SwiftUI 的学习重点并不是“少写代码”。真正重要的是理解：

- 哪些 view 被认为还是同一个 view。
- 哪些状态会被保留。
- `ForEach id` 是否表达了真实业务身份。
- 页面语义整体变化时，是否需要新的 identity。
- 什么时候应该用 `.id(...)` 强制重建。

UIKit 要求你管理对象，SwiftUI 要求你管理身份。

## SwiftUI 和 UIKit 的取舍

SwiftUI 的优势很明显：

- 写简单页面很快。
- 状态绑定减少了大量胶水代码。
- 表单、设置页、轻量工具界面非常舒服。
- 小型 macOS/iOS 工具项目开发效率高。
- UI 结构更接近最终界面描述。

UIKit 或 AppKit 的优势也很明显：

- 生命周期明确。
- 复用机制明确。
- 调试路径直接。
- 精细滚动、复杂手势、复杂列表更可控。
- 对性能和局部刷新有更强控制力。

如果只是做设置页、表单页、简单工具，SwiftUI 通常很合适。如果是复杂日历、富文本编辑器、拖拽排序、精细滚动控制、大量状态交叉的列表网格，UIKit/AppKit 的控制感会更强。

SwiftUI 适合表达 UI，UIKit 更适合控制 UI。

## 给新学习者的建议

如果你刚开始学习 SwiftUI，不要把它理解成“自动刷新 UI 的魔法”。它确实能让很多代码变短，但需要你认真理解 identity 和状态生命周期。

一些实践建议：

- 不要轻易用数组下标作为长期 identity。
- `ForEach id` 应该表达业务身份，而不只是当前位置。
- `@State` 不是普通属性，它依附于 view identity。
- `List`、`LazyVGrid`、`LazyVStack` 都可能带来复用和状态残留。
- 当页面语义整体切换时，可以考虑给外层容器加 `.id(...)`。
- cell 的关键显示状态最好基于当前上下文实时计算。
- 不要只跑编译验证 UI bug。
- 用户给出明确复现路径时，必须按同样路径启动 App 截图确认。
- 如果代码级验证通过但 UI 仍然错误，优先怀疑 identity、状态复用和懒加载容器。

这不是说 SwiftUI 不可靠，而是说 SwiftUI 的抽象层级更高。抽象层级越高，越需要知道它什么时候帮了你，什么时候挡住了你。

## 这次修复后的验证方式

这次问题最后真正确认修复，是因为做了完整的真实界面验证：

- 先清理旧的 App 实例，避免 Xcode、DerivedData、临时构建产物同时运行。
- 用 fresh 模式启动最新构建产物。
- 通过系统窗口 ID 只截取 App 主窗口，而不是截全屏。
- 人工确认 2026 年 7 月视图中，7/1 到 7/31 为正常深色，6/29、6/30、8/1、8/2 为淡化灰色。

这也提醒我：编译通过只能证明语法和类型正确，不能证明 UI 行为正确。尤其在 SwiftUI 中，视觉状态、identity 和懒加载复用相关的问题，必须通过真实界面验证。

## 结尾

SwiftUI 降低了 UI 代码的书写成本，但没有消灭 UI 系统本身的复杂度。它只是把 UIKit 中显性的生命周期、复用和刷新控制，转移成了隐性的 identity、diff 和状态管理。

对新手来说，SwiftUI 入门更快。对复杂界面来说，SwiftUI 的理解成本并不低。

真正的工程经验不是简单选择某个框架然后相信它，而是知道它的抽象边界在哪里。什么时候顺着它写，什么时候必须主动给它明确的身份、状态边界和验证路径。

代码变短了，不代表心智模型消失了。
